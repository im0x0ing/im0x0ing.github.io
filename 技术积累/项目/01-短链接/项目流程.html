短链接生成

用户输入长链接,自动生成短码.用户访问短链接,系统跳转到长链接.

  • 系统自动分配短码
  • 用户自定义短码
  • 同一长链接处理:重复生成返回已有短码

1. HTTP POST /v1/create_url
   ↓
2. server/route.go 监听到请求并进行路由分发
   ↓
3. controller/sequence.go → CreateLink()
   ↓
4. model/short_url/links.go → CreateLinks()
   ├── 4.1 GetLinksByUrl() → 检查是否已存在
   ├── 4.2 系统分配模式：
   │   ├── sequence.GetBorrowOrder() → Redis INCR
   │   └── tool.TenToAny() → Base62编码
   └── 4.3 MySQL Insert → 持久化
   ↓
5. 返回短链接：http://domain/{keyword}

短链接跳转

  • 根据短码查询对应长连接
  • http301永久重定向
  • 404错误处理
1. HTTP GET /{keyword}
   ↓
2. HTTP服务器接收请求
   ↓
3. CORS中间件执行（已注册的中间件）
   ↓
4. Middleware中间件执行（已注册的中间件）
   ├── 检查路径：包含"create_url"？→ 否
   ├── 提取 keyword（去除"/"）
   ├── model/short_url/links.go → GetLinksByKeyword()
   └── MySQL Select
   ↓
5. 301 Redirect → 长链接

结构

base 基础设施层

  • config数据库配置管理(从 .ini 配置文件读取配置，解析并映射到结构体)
  • db数据库连接 管理连接池和数据库客户端获取接口
  • 以及一些工具函数
    • 日志管理 zap+lumberjack滚动日志
    • 雪花算法
    • 进制转换

conf 配置文件目录

会被base层读取,包括数据库配置和工具配置

controller 控制器层

  • 控制器基类(一个容器,装所有功能)
    • 依赖注入接口(也就是说 这个容器里的功能并不是直接实现的,而是调用某个接口的方法.这样之后如果要换业务比较方便,不用修改基类代码)
    • 单例模式初始化(整个程序只有一个容器实例，不会重复创建)
  • 短链接创建控制器
    • 解析 post create url 表单参数
    • 调用CreateLinks(在model业务层)处理请求
  • http响应封装
    • 成功响应封装
    • 失败响应封装

model 业务逻辑层

  • 序列生成器模块
    • GetBorrowOrder()：使用Redis INCR生成全局唯一自增ID
  • 短链接业务模块
    • 数据模型结构体
    • 创建短链接业务CreateLinks 为长连接生成短链接
      • 判断长连接是否已存在
      • 系统分配 Redis INCR → Base62编码
      • 自定义
      • 持久化到mysql
      • 添加到布隆过滤器bloomFilter.Add(keyword)
      • 返回短链接
    • 根据长连接查询GetLinksByUrl 检查这个长链接是否已经创建过短链接了
    • 根据短链接查询GetLinksByKeyword 通过短码查找对应的长链接（跳转时用）
用户创建短链接
  • 长链接：https://www.google.com
  • 模式：1（系统自动分配）
    发送到服务器：POST /v1/create_url
  1. 然后解析完之后业务层CreateLinks() 函数开始工作,会调用GetLinksByUrl.如果已经创建过,就直接返回旧短码.
  2. 没有创建过,那么就需要开始创建,根据模式进行创建.
    1. 如果是系统分配,会order = Redis.INCR()获取唯一id,然后keyword = TenToAny(order)转换成62进制
    2. 如果是自定义,就先验证是否被占用,只要没被占用就可以使用用户自定义的短码
  3. 保存到数据库Insert(links)并 添加到布隆过滤器bloomFilter.Add(keyword)
redis

独立的数据库服务.可以在本地运行/远程服务器运行/网络连接访问
go程序构建redis命令,然后通过tcpip发送到redis服务器,redis收到命令后操作完返回结果.
redis是内存数据库(存储在独立服务器的内存),mysql是硬盘数据库(存储在服务器的硬盘)

用户访问短链接
  1. 用户在浏览器输入：http://127.0.0.1:8099/b
  2. server.Middleware()拦截请求,提取到短码
  3. GetLinksByKeyword(“b”)
    1. 查布隆过滤器.如果false,那么一定不存在.直接返回404,不查数据库
    2. 如果true,那么可能存在,进入第二层防护：SingleFlight
      1. 合并并发请求.第一个请求查数据库,如果查到了那么返回结果并且给server层处理结果进行重定向或者404
      2. 其他并发等待第一个共享结果
  4. c.Redirect(301, “https://www.google.com”) 重定向到原本的长连接,浏览器可以跳转.
布隆过滤器

组成:

  1. 一个很长的二进制向量(位数组),每一位只有0或1,初始化时全是0.每个短链接一起共用.如果一个短链接来进行test的时候,算出来的哈希函数值对应的位不是1,那么说明肯定不在(因为一开始add的时候都会置1),那如果是1,也不能证明一定在,但是既然可能在,就继续查询mysql.
  2. k个哈希函数,可以将输入映射到位数组的某个索引上
singleflight

s.sfGroup.Do(key, fn)。这个库 (golang.org/x/sync/singleflight) 的源码非常短，核心思想是 “复用” 和 “阻塞等待”。
SingleFlight 的 Group 结构体内部维护了一个 Map：

do: 当10000个请求同时调用do时

  1. 进门先抢group的互斥锁
  2. 判断身份 如果是第一个人
    1. 查map,发现key不存在
    2. 创建一个新的call对象
    3. call.wg.Add(1)
    4. 把key→call放入map
    5. 释放锁mu
    6. 执行真正的fn()查数据库
  3. 如果是后来者
    1. 查map发现key已经存在了,
    2. 那么拿到call对象的指针,释放锁mu
    3. 原地阻塞睡觉,等待leader环形
  4. leader干完活了把结果写到call.val
  5. leader再次加锁删除key
  6. 调用calldone
  7. follower接触阻塞
  8. 所有follower直接读取call.val
  9. 拿着同一个结果返回了

server http服务层

  • http服务器启动
    • 启动Gin HTTP服务器并配置超时时间\最大请求头大小,监听来自浏览器的请求
  • 路由配置;定义不同请求应由谁处理
    • route()注册路由
    • 设置cors跨越中间件
      • 允许来自不同网站的浏览器访问这个服务器，防止跨域访问被浏览器阻止
    • 设置短链接跳转中间件Middleware
      • 如果是创建短链接就继续处理(这里在下面api注册路由的时候,会分发到CreateLink函数的)
      • 如果是短链接跳转 这里负责调用GetLinksByKeyword或者返回404
    • 登记api路由: 根据:路由组(v1)\http方法(.post)和url路径,调用对应处理函数.程序启动时注册(main中server.run run中route().) → 运行时查找路由表 → 调用处理函数.()
