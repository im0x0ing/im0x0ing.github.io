优化总览

优化总览

Jan 19, 20266 min read

核心思想

  1. 用户输入一个很长的网址（比如：https://www.google.com/search?q=hello）
  2. 系统生成一个短码（比如：abc123）
  3. 用户访问短链接（比如：http://你的域名/abc123）
  4. 系统自动跳转到原始长网址
  • 当我们在浏览器里输入 https://dwz.cn/XzhYJMkZ
  • DNS首先解析获得 https://dwz.cn/ 的 IP 地址
  • 当 DNS 获得 IP 地址以后，会向这个地址发送 HTTP请求，查询短码 XzhYJMkZ
  • https://dwz.cn/ 服务器会通过短码 XzhYJMkZ 获取对应的长 URL
  • 请求通过 HTTP 301 转到对应的长 URL http://www.baidu.com

项目结构

  • main.go 程序入口 启动整个服务

  • base 基础层 包括初始化配置 数据库 工具

  • conf 配置文件

  • controller 控制器层 处理http请求

  • model 模型层 业务逻辑 包括序列生成器和短链接业务逻辑

  • server 服务器层,包括路由和http服务

  • controller: 只负责接收http请求和解析参数

  • service 负责核心逻辑 包括雪花算法生成和布隆过滤判断

  • model 只负责存取数据库/redis

    • Controller 调 Service。
    • Service 调 Model。
    • 不能让 Model 去调 Controller（这就是循环引用，Go 编译器会报错）

GORM与XORM

ORM object relational mapping
用操作结构体的方式来操作数据库，而不需要复杂的手写SQL语句

“原来的 Baseline 使用 XORM，我考虑到在高并发场景下，复杂的 ORM 反射带来的性能开销较大。我对比了 GORM V2 的性能改进，并最终决定在核心读写路径上使用原生的 sqlx 或 sqlc，而在后台管理界面使用 GORM。这种‘读写分离、工具分离’的设计更符合高性能场景。”

服务

gin web框架

gin是go语言里面最流行的web框架.

  1. 当用户在浏览器输入网址请求,gin负责接收请求
  2. 解析参数
  3. 把结果打包发回给用户.
  4. 替代了手写解析http协议哪些繁琐的底层代码

mysql 数据库

关系型数据库 负责永久存储数据

为了把长链接存进数据库,我们需要给它一个唯一id,然后把这个id转成短码.如果用自增id,容易被猜出总共有多少数据,而且如果有多个数据库,id就会重复.

snowflake

twitter发明的一种算法

  1. 不重复: 利用时间戳+机器编号+序列号生成一个64位整数,可以做到全球唯一
  2. 有序: 并且总体上是随时间递增的,对数据库索引比较的友好.
  3. 高性能: 而且本地生成,不需要联网查询数据库,速度很快.
    符号位+41位时间戳（毫秒）+10位机器id+12位序列号（同毫秒内序列）
  • 符号位始终为0
  • 41位时间戳可用69年（基于设定的starttime的相对时间）
  • 支持1024台机器
  • 同一毫秒内可生成4096个id

redis 缓存

内存数据库 速度极快比mysql快几万倍,但是断电可能会丢失部分数据,所以主要当作缓存在使用.
因为短链接业务是读多写少,生成一次,可能会被几万人点击访问,如果都去查mysql会被挤爆炸,所以引入redis来抗并发

布隆过滤器 bloom filter

如果黑客疯狂请求一堆不存在的短链接,redis查不到mysql也查不到,大量的空请求直接打穿redis压在mysql上,数据库直接挂掉,叫做缓存穿透.

是一个节省内存的数据结构,守在最前面,如果他说不在,那就肯定不在,把请求驳回,不查询redis和mysql了.

SingleFlight

如果某个短链接同一秒钟有一万人点击,刚好这时候redis里面这个链接的缓存过期了,那么又达成了缓存击穿.

singleflight是go官方的一个库,当一万个请求同时来的时候,singleflight会挡住他们,只放第一个人查询数据库
