第一维度：架构设计与ID生成（必问区）

  1. Q: 为什么选用Snowflake（雪花算法）？相比UUID 和数据库自增ID，它的优势在哪？
    其他唯一id生成法
    UUID

    基于mac地址时间戳和随机数生成的128位字符串,太长而且无序,而且转换成62进制还是很长

    美团leaf

    Redis 批量预取的升级版。它直接从数据库（MySQL）批量拉取 ID 号段缓存到内存。它使用了“双缓存”技术。当内存中第一个号段用掉 10% 时，它会异步去加载下一个号段，保证业务取号时永远不需要等待网络 IO。

    百度uidgenerater

    也是雪花算法的变种，但它支持“借用”未来时间。如果当前秒的 ID 用完了，它可以预支下一秒的。

    Link to original
  • UUID 太长不适合短连接 且无序（影响 数据库的B+ 树写入性能）,插入位置随机,会导致页分裂
  • 自增 ID 容易被遍历（不安全）且分库分表麻烦,每次生成要访问数据库
  • Snowflake 有序、纯数字、高性能、天然支持分库分表.本地生成不依赖网络。

第二维度：缓存体系与布隆过滤器（高危区）

针对你简历中的：“引入布隆过滤器拦截99% 无效请求，解决缓存穿透”

  1. Q: 布隆过滤器的原理是什么？为什么它能判断“一定不存在”？

    • 关键点： 多个 Hash 函数，位图（BitMap）。如果某个位置是 0，则一定不存在；全为 1，则可能存在（误判）。

  2. Q: 既然有误判（False Positive），如果布隆过滤器误判了（说存在，实际DB 里没有），你的系统会怎么样？

    • 回答逻辑： 误判会导致少量请求穿透到 DB，但因为布隆过滤了绝大多数，DB 压力可控。DB 查不到时，返回“链接不存在”给用户。

  3. Q: 短链接如果过期了或者被删除了，布隆过滤器不支持删除怎么办？

    • _高频考题：_布隆过滤器很难删除。

    • 解法： 1. 忍受误判（靠 DB 兜底）。2. 使用布谷鸟过滤器（Cuckoo Filter）支持删除。3. 定期重建布隆过滤器。

  4. Q: 你的Redis 缓存过期时间是怎么设置的？有没有考虑热点Key 过期导致的“缓存雪崩”？

    • 解法： 随机过期时间（Random TTL），避免同一时刻大面积失效。

第三维度：并发控制与SingleFlight（源码级拷打）

针对你简历中的：“利用Go SingleFlight 合并热点链接瞬时并发读请求”

  1. Q: SingleFlight 是怎么做到合并请求的？ （源码逻辑）

    • 关键词： map[key]*call，sync.WaitGroup。

    • 流程： 第一个请求进来，创建一个 call 结构体放入 map，并加锁；后续请求发现 map 里有这个 key，就 wg.Wait() 等待；第一个请求执行完，wg.Done()，结果共享给所有等待的请求。

  2. Q: 如果那个“领头”的请求（第一个请求）挂了或者超时了，后面的请求怎么办？

    • _深坑：_可能会导致所有等待的请求一起死锁或超时。

    • 解法： 使用 DoChan 配合 select 超时控制，或者在 Do 内部逻辑里做好超时处理。

  3. Q: SingleFlight 适合什么场景？写请求能用吗？

    • 回答： 只适合读场景（且是幂等的读）。写请求不能合并，否则数据就乱了。

+1

  1. Q: 你的MySQL 表结构是怎么设计的？

    • 关键点： short_code (主键/唯一索引), original_url, created_at, expiration_time。

    • 索引： short_code肯定要加索引。

  2. Q: 你说解决分库分表冲突，那你具体是怎么分库分表的？分片键（Sharding Key）是谁？

    • 逻辑： 如果用 short_code 查，就用 short_code 做分片键。如果是 Snowflake ID，可以直接模运算分表。

  3. Q: 什么是覆盖索引？在你的短链查询里怎么用到？

    • 场景： 如果 SQL 是 select original_url from table where short_code = ?，可以给 (short_code, original_url) 建联合索引，这样就不用回表了。

第五维度：极端场景与系统稳定性（压力面）

  1. Q: 如果Redis 整个挂了，你的系统会崩吗？怎么降级？

    • 回答： 开启 Redis 的高可用（Sentinel/Cluster）。如果真全挂了，使用本地缓存（Go sync.Map 或 BigCache）做一级缓存，限流保护 DB。

  2. Q: 同一个长链接生成多次，是返回同一个短链，还是生成新的？

    • 业务逻辑题：

      • 方案A（省空间）：查表，有了就返回旧的（需要给长链建索引，写性能低）。

      • 方案B（高性能）：直接生成新的（会有空间浪费，但也是一种选择）。看你项目怎么实现的，自圆其说即可。

  3. Q: 恶意用户疯狂请求不存在的短链接（攻击），你的系统怎么防？

    • 回答： 这就是缓存穿透，我的布隆过滤器就是干这个的！配合 Nginx 限流（Rate Limiting）。

  4. Q: 你的QPS 大概是多少？你是怎么压测的？

    • 数据： 别瞎编。单机 Go + Redis 轻松上万。

    • 工具： 即使没测过，也要说：“我使用 wrk 或 JMeter 进行压测…”
