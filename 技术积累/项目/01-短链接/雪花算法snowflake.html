雪花算法snowflake

雪花算法snowflake

Jan 21, 20267 min read

当前方案与问题

  • 使用 Redis INCR 生成自增 ID
  • 通过 Base62 编码将数字 ID 转换为短码

性能瓶颈:每次生成id都要访问redis,网络延迟影响性能[一次 Redis 请求往返（RTT）通常需要 1-5 毫秒。]
redis故障会导致id生成失败
无法支持分库分表: 数据量太大会需要分库分表,但是如果用自增id,在两个数据库之间很难索引了.

中心化:所有服务器节点都向同一个redis要id

扩展性差,高并发下redis成为瓶颈,单台redis的吞吐量是有上限的,而且处理一个自增id任务,是无法用cluster集群去做的,只能绑定在一个key也就是一台机器上.

优化

redis去批量预取
雪花算法:目前分布式系统最主流的方案:完全去中心化

  1. 本地生成：无需访问 Redis，性能提升 100+ 倍
  2. 分布式支持：每个实例独立生成唯一 ID，支持分库分表
  3. 高可用性：不依赖外部服务，单机也能工作
  4. 性能提升：从 Redis 网络调用（1-5ms）提升到本地内存操作（<1μs）
其他唯一id生成法
UUID

基于mac地址时间戳和随机数生成的128位字符串,太长而且无序,而且转换成62进制还是很长

美团leaf

Redis 批量预取的升级版。它直接从数据库（MySQL）批量拉取 ID 号段缓存到内存。它使用了“双缓存”技术。当内存中第一个号段用掉 10% 时，它会异步去加载下一个号段，保证业务取号时永远不需要等待网络 IO。

百度uidgenerater

也是雪花算法的变种，但它支持“借用”未来时间。如果当前秒的 ID 用完了，它可以预支下一秒的。

实现

  1. 启动时.init :NewSnowflakeGenerator()
  2. 尝试从redis抢占机器id
    1. 成功 启动心跳续期
    2. 失败 启动本地ip降级策略
  3. GetBorrowOrder,优先使用Snowflake.GetId(),如果不可用,降级到redisincr自增
  4. 退出的时候,收到信号,就调用sequence.Release,释放redis中的机器id

有两次降级处理:

降级策略1：Redis 机器ID分配失败 → 使用本地IP计算机器ID

触发条件：

  1. Redis 连接失败（网络问题、Redis宕机）
  • acquireMachineID() 中的 r.SetNX() 返回错误
  1. 所有机器ID都被占用（1024个实例都在运行）
  • 遍历 0-1023 都失败，返回 “all machine IDs are occupied”

降级后的行为：

  • 仍使用 Snowflake 算法生成ID

  • 机器ID通过本地IP的hash值计算（0-1023）

  • 不再启动心跳续期（因为Redis不可用或不需要）

  • 不同IP的hash值取模1024后可能相同（概率约 1/1024）

  • 一台用Redis分配，一台用本地IP，机器ID可能相同
    但是时间戳和序列号一般不同,想要同时相同还是很有难度..
    但是k8s的时候,K8s 可以在一秒内拉起几十个 Pod。如果这些 Pod 跑在同一台宿主机上，它们看到的系统时钟是完全一致的。每个 Pod 的序列号（Sequence）在内存里都是从 0 开始的。那么,要想id差异化就得全部依赖机器id.那么就换一些复杂的哈希算法,以及加入hostname(pod名称自带唯一后缀)

回拨: 小于阈值睡眠,大于阈值报错,并且兜底:就是直接强行增加时间戳.

降级策略2：Snowflake 初始化完全失败 → 使用 Redis INCR

触发条件：

  • NewSnowflakeGenerator() 返回非 nil 错误
  • 当前实现中几乎不会发生
    降级后的行为：
  • 不再使用 Snowflake
  • 完全回退到 Redis INCR 自增

雪花算法的问题

  • 时钟回拨 如果机器时间回退,那么通常只能报错或者死等,直到追回正常时间.
  • 机器id管理,在Docker/k8s的情况下,容器是动态销毁和启动的,如何保证拿到不重复的workid

如何保证? 容器启动的时候,去redis抢占一个还没被占用的id,走的时候释放掉
或者利用k8s部署的时候,手动给每个实例注入不同的workid变量

  • id规律性太强 能够直接反推出生成时间

问题

  1. 为什么你觉得 Snowflake ID（19位）和 Redis ID（从1开始）不会冲突？如果项目运行了很久，Redis ID 也长到了 19 位呢？
  • 这是一个数学规模问题。Redis ID 要达到 19 位（即 10 的 18 次方级别），即便每秒生成 100 万个 ID，也需要 3 万年 才能达到这个数字。
  • 而 Snowflake ID 因为起始时间戳的存在，天生就从 10 的 18 次方左右开始。因此，在可预见的几千年内，两者的数值空间是互不重叠的。
  1. 如果redis运行中挂了,心跳续期失败怎么办?别人会抢占这个id.但是机器id都是一开始运行就申请好,所以原本的也不会发现
